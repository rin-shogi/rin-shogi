# ITERATION: 棋力向上イテレーションの運用フロー

plan の Phase F(M6)以降の運用ループ。本端末完結で R3300 帯を狙う段階で、
**何をどの順で試し、何を採否判断の基準にするか** を明確化する。

## 基本サイクル

```

└──────────────┬─────────────┘
               │
               ▼
┌────────────────────────────┐
│ 2. 改善仮説を1件選ぶ        │  下記「改善軸の優先順位」から
└──────────────┬─────────────┘
               │

               ▼
┌────────────────────────────┐

               ▼
```

## 改善軸の優先順位(D13 に基づく)

本端末段階での効率順:

### 1. 定跡(F2)— 最優先

**期待効果**: 数十 Elo の上積み(序盤の手損を直接防ぐ)
**コスト**: 低(計算資源不問、定跡 DB 構築は数時間〜)
**やり方**: `scripts/build_book.py`

- 公開定跡(やねうら王同梱・技巧定跡・Apery定跡 等)の取り込み・統合
- 評価関数で定跡の各手を再評価(深い探索で「悪い」と判定された手は外す)
- 自己対戦で頻出する局面に手を追加
- 序盤戦型ごとの定跡精度確認(振り飛車・角換わり・矢倉 等)

### 2. 探索パラメータ自動チューニング(F3)— 中

**期待効果**: 数〜十数 Elo
**コスト**: 中(SPSA で 1パラメータ ≒ 数百局必要、数日〜)
**やり方**: `scripts/tune_search.py`(SPSA / OptunaベースSPSA)

- 時間配分(`MoveOverhead`, `EngineNumThreads` 等)
- 置換表サイズ(`USI_Hash`, `Hash`)
- 主要枝刈り係数(LMR, NullMove, Futility 等のやねうら王設定可能パラメータ)
- 注意: SPSA は試行回数が多いので、本端末では「重要パラメータ 3-5 個」に絞る

### 3. 教師データ・学習レシピの本端末規模調整(F4)— 限定的

**期待効果**: 本端末規模では数 Elo(計算資源不足、限界がある)
**コスト**: 高(教師生成 数十時間、学習 数時間〜)
**やり方**: `scripts/gen_teacher.py` + `scripts/train_nnue.py`

- 自己対戦の depth(教師の質)
- floodgate 棋譜の混合比率
- 学習レシピ(eta, batchsize, loop)
- **本端末で出発点を超えるのは難しい想定**(R8、Phase G で再挑戦)

### 4. 戦型特化(F5)— ニッチ

**期待効果**: 局所的(特定戦型で数 Elo)
**コスト**: 低-中(定跡追加 + 重要局面の評価関数微調整)
**やり方**: `scripts/build_book.py` で戦型特化定跡を追加

## 改善仮説の書き方(`results/iteration/HYPOTHESES.md` に蓄積)

```markdown
## H001: 矢倉系の定跡を 50局面追加

- 軸: 定跡(F2)
- 仮説: 矢倉系で序盤に評価値を落としているので定跡で防げるはず
- 比較: ベースライン(現行 evals/baseline/suisho + 既存定跡) vs 追加版
- 試合数: 200局
- 期待: 勝率 53% 以上(統計有意)
- 結果: <測定後に記入>
- 採否: <記入>
```

## ベースライン管理

- `evals/CHANGELOG.md` の最上行が常に「現行ベースライン」
- 採否判断は `match_summary.py` の `Statistically significant: YES` を必須とする
- 採用版は `evals/trained/v<version>/` または `evals/book/<version>/` に固定し、CHANGELOG に追記

## イテレーションのリズム

| サイクル単位 | 期間 | やること |
|---|---|---|
| デイリー | 数時間 | 仮説 1 つを試す |
| ウィークリー | 1 週 | floodgate レーティングのレビュー、ベースライン更新の判断 |
| マンスリー | 1 ヶ月 | F2-F5 のどれに重点を置くかの戦略レビュー、Phase G 移行判断材料の集計 |

## 失敗時の対応

| 状況 | 対応 |
|---|---|
| 連続 5 仮説が NO | 改善軸の見直し(優先順位を入れ替え)、ベースライン自体の問題を疑う |
| 一見改善したが floodgate で出ない | 自己対戦の偏り(時間制・対戦相手のバリエーション不足)を疑う、N を増やす |
| 急激に悪化 | 直近変更を rollback、`evals/trained/` の前バージョンに戻す |
